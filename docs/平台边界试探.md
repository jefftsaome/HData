# 平台边界试探档案

> 目的：长期维护平台各项边界的实测记录——哪些点测到什么程度
> 确认安全、哪些点触到了边界、哪些还是未知。
>
> 维护约定：
> 1. 每条记录必须有**日期 + 测试方法 + 样本/证据**，不臆测；
> 2. 状态分三档：✅ 已测安全（含测到的程度）｜⚠️ 已触边界
>    （平台明确拒绝/惩罚）｜❓ 未知（未测或证据不足）；
> 3. 生产配置一律取"实测安全值并留余量"，禁止贴着边界运行；
> 4. 每次边界实验（无论成功失败）都追加登记，过期结论复核后更新。

## 2. WS 连接层（详录）

### 2.1 同 IP 新建连接速率 ⚠️ 已触边界

- **方法**：`scripts/streak_hunter.py` 首次活测，10 个监控分片以
  ~1.5s 间隔连续建连
- **现象**：第 6~7 条握手返回 HTTP 403；随后 ~10 分钟内任何新连接
  均 403（含探针单条建连）；已建立连接不受影响
- **推论**：WAF 对同 IP 新建 WS 有速率/窗口限制，触发后短封
- **安全值**：**3s/条**（实测 3s 间隔连建 10 条全部成功）
- **工程对策**：`TableMonitor.__aenter__` 内置 3s 间隔 + 3 次
  指数退避（5/10/15s）+ 失败剔除降级

### 2.2 同 IP 并发连接总数 ✅ 已测安全（下限）/ ❓ 硬上限未知

- **方法**：`scripts/probe_ws_cap.py`，3s 间隔逐条建连
- **记录**：7 条（1 大厅 + 6 监控）全成功；11 条尝试中 10 成功、
  1 条传输层失败（非 403，疑似软限制，未重试）
- **注意**："11"是账号数量天花板（1 账号 1 连接），**不是平台
  上限的实锤**；真实硬上限需 >11 个账号压测（待办 #A）

### 2.3 单连接监控桌数 ✅ 已测安全（下限）/ ❓ 上限未知

- **方法**：streak_hunter 活测（--min 3，6 分钟）
- **记录**：110 桌分布在 10 条连接，单连接 11 桌，事件流、
  进桌、被踢重进全部正常；单连接桌数上限未触到

### 2.4 单账号同时连接数 ⚠️ 边界 = 1

- **方法**：实测 + jti 机制分析（见《平台接入机制》§3）
- **结论**：同一账号同时只允许 1 条游戏连接；新登录/刷新 token
  后旧连接被 10026 顶掉。横向扩容只能加账号，不能加连接

### 2.5 token 绑定登录 IP ⚠️ 实锤绑定

- **方法**：`scripts/probe_proxy.py` 对照实验——强制全量重登取
  新鲜 token；A 组直连 WS 成功；B 组同一 token 走代理 WS → 10026
- **结论**：登录/刷新/WS 必须全程同一出口 IP；账号经代理必须
  粘性绑定（详见《代理接入》§3）

### 2.6 心跳 ⚠️ 无心跳 40~60s 断连

- 协议心跳 pid=3、10s 间隔（与官方前端一致）可稳定保活；
  无心跳 40~60s 服务端主动断连（实测）

## 3. 游戏行为层

### 3.1 连续未下注 ⚠️ 平台规则边界

- 连续 3 局未下注：服务端推 123 预警（noticeId=21002），不踢人
- 连续 5 局未下注：推 102 离桌（leaveTableType=2，noticeId=21003）
- 被踢后重发进桌指令可重新进桌；进桌指令**不能**预防踢出（实测）
- `kick_policy="stay"` 自动重进已验证（6 分钟活测无中断）

### 3.2 大厅持续订阅 ✅

- 10089 一次订阅后 ~1.4s/帧持续推 10052（90s 采样 64 帧、
  1124 条人数更新）；数小时级长跑稳定性待 24h 测试（待测 #E）

### 3.3 掉线删失原因细化（探测性重进）🔬 机制上线，待样本（2026-07-21）

- **背景**：此前掉线一律记 censored_disconnect，无法区分"我方网络
  问题"与"平台踢出且封锁桌台"——后者疑似平台把桌台单独挂起来
  点杀在场用户，是重点监测对象
- **机制**（streak_hunter `_probe_pending`）：事件流中断 → 全部
  episode 先记 censored_disconnect 兜底 → 监控重建成功后逐桌
  探测性 add_table，以 12s 内 401 快照落地为"确实进桌"判据：
  - 能进 → 改标 **censored_network**（桌台可进，是我方网络问题），
    随即离场交还发现层按常规流程重新发现（不续接旧 episode）
  - 进不去/超时 → 改标 **censored_kick**（疑似平台踢出且封锁）
  - 人为 Ctrl+C → **censored_manual**；进程强杀遗留 → 保持
    censored_disconnect（原因不可知）
- **局限**：判据是"能否重进"，无法区分"平台封锁"与"桌台刚好
  满员/维护/换靴清桌"；出现 kick 样本时需人工复核当时大厅状态
- **状态**：历史 1429 条 censored_disconnect 无法回溯区分，只能
  向前积累；kick 样本出现时登记到本节

## 4. 登录与凭证层

### 4.1 连续打码登录 ✅

- 11 个账号连续打码登录（每个 ~10s），未触发频率限制

### 4.2 缓存会话有效期 ⚠️ ~10 小时

- 站点会话（cookies 等）约 10 小时后失效，缓存命中但刷新失败，
  需走完整打码重登；game_token 本身有效期更短，靠刷新维持

### 4.3 打码平台 ⚠️ 偶发失败

- geepass/jfbym 识别偶发失败（识别错误/网络错误），内置 3 次
  重试可兜住；成本约 ¥x/次（按打码平台定价）

### 4.4 站点会话不绑 IP ✅（2026-07-20）

- **方法**：streak_hunter 代理活测（日志 `data/proxy_test_run.log`）——
  账号持本机缓存的站点会话（cache hit），经**代理出口**调
  game JWT 刷新接口
- **结果**：4 个缓存账号（lshaoxia1/linbing1/lds001/lds002）全部
  `game JWT refresh succeeded`，随后 WS 经同一代理建连成功
- **结论**：**只有 game_token 绑 IP（见 2.5），站点会话（cookies）
  不绑 IP**。换代理出口时无需强制打码重登——经新出口刷新一次
  game_token 即完成 IP 换绑；打码重登仅在站点会话本身过期
  （~10h，见 4.2）时才需要

### 4.5 game JWT 刷新接口速率限制 ⚠️ 已触边界（2026-07-20）

- **方法**：streak_hunter 直连活测，11 账号缓存全部命中、登录秒回，
  建连前刷新以 3~4s 间隔密集发出（日志 `data/streak_hunter.log`
  18:05 段）
- **现象**：前 5 次刷新成功（18:05:06~13），第 6 次（18:05:17，
  第 3 个分片建连时）被拒
- **对照**：2026-07-19 冷缓存跑（打码登录天然把刷新隔开 10s+）
  与 2026-07-20 代理跑（2 IP 分摊刷新密度）均未触发
- **推论**：刷新接口对同 IP 有速率限制，量级约 5 次/10s
  （**精确阈值未测定**）；冷启动批量建连是主要触发场景
- **工程对策**（已落地）：进程级节流（≥2s/次串行排队）+
  60s 新鲜跳过（登录刚刷过的 token 建连前不重复刷）+
  失败退避 5s 重试一次再兜底重登
- **附注**：本次事故同时暴露 `_refresh_cb` 兜底重登错用主账号的
  bug（分片刷新失败会重登主账号并把主会话塞进分片，多连接
  同账号 jti 互顶形成打码死循环），已修复并补 6 例回归测试

## 6. 桌台推送按账号分组（#16 详录）

- **发现日期**：2026-07-20（用户用 linyan1 跑 verify_table 找不到 H38 触发排查）
- **方法**：linyan1 实时 `get_tables()` vs hunter 11 账号（lds001-008 /
  linbing1 / lshaoxia1 / sushizhen1，同批次）采集库 `lobby_snapshots` 对比
- **实测**：
  - hunter 11 账号（同批次）可见 **227 桌**；linyan1 仅 **47 桌**；
    交集仅 **41 桌**；linyan1 另有 6 桌是 hunter 从未见过的
  - hunter 组可见 `经典百家乐B12`（table_id=2658），linyan1 不可见；
    hunter 组与 linyan1 组均不见 `H38`（H3x 整段缺失）
  - 同一账号多次登录可见池稳定，非动态负载调整
- **推测（未证实）**：桌名前缀字母（B/C/H/M/U/J/R/K/D/A）疑为
  厅/代理线标识，账号注册渠道决定归属组；分组可能承载商业归属
  （代理分润）功能，不能排除差异化管理
- **影响**：
  1. 监控覆盖率天花板由账号组决定；扩覆盖需不同批次的账号
  2. 新账号接入前应先用 `verify_account.py` 拉列表对比可见桌集
  3. 数据分析应按组切片，避免异质样本混合（见分析报告第五节）
- **状态**：⚠️ 机制已确认存在；分组规则（注册渠道？邀请链？）未测
