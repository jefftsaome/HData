# 极验4（GeeTest v4）验证码破解与排查

> 本文写清极验 4 文字点选验证码的完整破解链路，以及**平台日后更新导致失效时
> 如何定位是哪一环坏了**。配合《加解密逻辑全景.md》阅读。
> 数据基于 2026-07-18 代码；标注 ⚠️ 的都是平台更新后最可能变化的点。

---

## 0. 一句话总览

登录 = 域名解析 → 验证码预注册 → 极验 load（拿挑战图）→ 打码平台识别坐标 →
极验 verify（提交 `w` 加密答案包）→ 站点二次校验 → login（拿站点 token）→
member/jwt → venue/launch（拿游戏 JWT）→ WS 连接。

双 token 架构：**站点 session token**（96 hex，`X-API-TOKEN`）+ **游戏 JWT**
（约 24h，从 venue/launch 返回的加密 params 中解出）。

破解验证码的三块拼图：**w 参数自生成**（RSA+AES，纯 Python）、**答案靠打码平台**
（geepass/jfbym）、**站点 API 签名靠官方 wasm**（X-API-XXX）。

## 1. 登录完整时序（逐步带请求细节）

### 步骤 0 — 域名解析（无固定主站 ⚠️）

- 入口种子：`https://leyu.com` / `leyu.me` / `hth.com`（`hdata/auth/domain.py:30-34`）；
- **必须用 Python urllib 拉入口站**（curl_cffi 的 TLS 指纹会被 ban，domain.py:5-7）；
- 三种解析格式：`mappings.set(...)`、`/code.js` 里的 `lypcurls = '...'`、
  兜底正则 `https://www\.[^"']+\.vip:\d+`（domain.py:192-206, 221）；
- 缓存 `.cache/domain.json`，TTL 30 分钟，命中先探活（任何 HTTP 响应含 4xx 都算活）；
- 产物形如 `https://www.5qk8bt.vip:3962` —— **真实域名自带非标准端口，小时级轮换** ⚠️；
- 全失败时回退环境变量 `HDATA_DOMAIN`。

### 步骤 1 — 验证码预注册 kaptchcate

```
POST {domain}/site/api/v1/user/member/kaptchcate
Body: {"kType": 4}                                   # 4 = GeeTest v4
Resp: {"status_code": 6022, ...}                     # 6022 = 成功 ⚠️ 不是 6000
```

失败不阻断（与浏览器容错行为一致）。代码：`hdata/auth/http_login.py:198-215`。

### 步骤 2 — 极验 load 拿挑战（`hdata/auth/captcha.py:30-59`）

```
GET https://bcaptcha.botion.com/load
  ?captcha_id=eaffad4f65a38a259ae369faf0c2f1a3     ⚠️ 硬编码3处：http_login.py:49 / captcha.py:19 / token_manager.py:507
  &challenge=<客户端随机 uuid4>
  &client_type=web&risk_type=word&lang=zh-cn
  &callback=geetest_<毫秒时间戳>
Headers: Referer: {domain}/user/login
```

JSONP 响应 `data` 字段：

| 字段 | 含义 |
|---|---|
| `lot_number` | 挑战批次 ID（32 hex），贯穿 verify → validateGeeCheckV2 → login.codeId |
| `payload` / `process_token` | 加密挑战数据 / 服务端会话 token，verify 原样带回 |
| `pow_detail` | `{"hashfunc":"md5","version":"1","bits":0,"datetime":"..."}`，w 的 PoW 用；**`bits` 必须存在**，缺了 `build_e_obj` 直接报错（geetest_signer.py:201-204） |
| `imgs` | 背景图路径（300×200 JPG，上有若干汉字），完整 URL = `https://static.botion.com/{imgs}` |
| `ques` | 3 张参考字图路径（64×65 RGBA PNG），即"题目"——按此顺序点击背景图中的对应汉字 |
| `captcha_type` | `"word"` —— **文字点选，不是滑块** ⚠️ 类型变了整条链路废 |
| `pt` / `payload_protocol` | 均为 `"1"`，verify 原样带回 |

> v4 与 v3 的区别：**没有 gt/challenge 两次 init**。`gt` 的角色由 `captcha_id` 取代
> （全代码无 `gt=` 残留）；`challenge` 是客户端自己 uuid4 生成的，不参与后续请求。
> botion 端点只有 **load** 和 **verify** 两个，都是 JSONP。

### 步骤 3 — 打码平台识别坐标

产物：`coords = "x1,y1|x2,y2|x3,y3"`（基于 300×200 原图像素）。详见 §4。

### 步骤 4 — 极验 verify（`http_login.py:106-191`）

```
GET https://bcaptcha.botion.com/verify
  ?callback=botion_<毫秒时间戳>
  &captcha_id=eaffad4f65a38a259ae369faf0c2f1a3
  &client_type=web
  &lot_number=...&payload=...&process_token=...
  &payload_protocol=1&pt=1
  &w=<1216 hex>                                       # 见 §3
Headers: Referer: https://www.leyu.me/               # 注意：写死 leyu.me
```

- 成功：`data.result=="success"` 且 `data.seccode` 含三件套
  **`captcha_output` / `gen_time` / `pass_token`**（代码强制校验三者非空，
  否则 `VerifyError("incomplete_seccode")`）；
- `captcha_output` 实测 512 base64url chars = 384 bytes 二进制
  ⚠️ 长度勿硬编码（历史值 312 bytes 已过时）；
- 失败：`data.result=="fail"` + `fail_count`（说明服务端已正常解密 w 进入判分，
  只是坐标不对——**fail 说明 w 没问题；其他错误才怀疑 w**）。

### 步骤 5 — 站点二次校验 validateGeeCheckV2（`http_login.py:227-269`）

```
POST {domain}/site/api/v1/user/member/validateGeeCheckV2
Body: {"validate_way": 1, "lot_number": "...",
       "captcha_output": "...", "gen_time": "...", "pass_token": "..."}
Resp: status_code==6000，data.captcha_args.user_ip = 服务端看到的出口 IP
```

`user_ip` 是计算 login 的 **X-API-FINGER** 的关键输入（见 §6）。

### 步骤 6 — login 拿站点 token（`http_login.py:272-323`）

```
POST {domain}/site/api/v1/user/login
Headers: + X-API-FINGER: <32 hex>                    # 仅 login 接口携带 ⚠️
Body: {"name": 账号,
       "password": <MD5(明文) 32hex 小写>,            # 纯 MD5 无盐
       "Kaptchcate": 0,                              # 0 = 验证码已校验过
       "codeId": <lot_number>}                       # 关联验证码批次
Resp: status_code==6000，data.token = 96 hex → X-API-TOKEN
```

### 步骤 7 — member/jwt 拿站点 JWT（`http_login.py:326-351`）

```
POST {domain}/site/api/v1/user/member/jwt   （带 X-API-TOKEN）
Resp.data = 站点 JWT → payload.uuid（exp 约 +7 天）
```

### 步骤 8 — venue/launch 拿游戏 JWT（`hdata/auth/session.py:264-329`）

```
POST {domain}/game/api/v1/venue/launch
Headers: 公共头 + X-API-XXX（wasm 动态签名，/game/api 强制校验，假签名 → 6003 ⚠️）
Body: {"enName": "YBZR"}                             # 场馆代码 ⚠️
Resp.data.url = "https://<游戏域>/?params=<base64>&ttl=<ts>&signature=..."
```

`params` 解密：**AES-128-ECB + PKCS7，key = (ttl + "AES")**（`params.py:28-78`），
解出 `{"playerId", "token"（游戏 JWT）, "backendDomainUrl", "backendDomainUrlList"}`。

### 步骤 9 — WS 连接（`session.py:433-481`）

```
wss://wsproxy.{backendDomainUrl}/?playerId=..&jwtToken=..&deviceId=..
     &platformId=1&applicationId=5&version=v1.0.5     ← 静态后缀缺失握手 HTTP 500 ⚠️
```

详见《加解密逻辑全景.md》§3.4。

## 2. w 参数生成（破解核心）

### 2.1 w 是什么

`w` = 加密的"答案包"，verify 请求的唯一动态参数。**总长恒 1216 hex = 608 bytes**。

生成公式（`hdata/auth/geetest_signer.py:275-279`）：

```python
random_key      = _rand_uid()                       # 16 个 hex 字符
encrypted_input = AES_128_CBC(plaintext=e_obj_json, key=random_key,
                              iv=b"0000000000000000")   # ⚠️ IV 是 ASCII '0'×16，不是 \x00×16
encrypted_key   = RSA_1024_PKCS1v1_5(random_key, geetest_public_key)
w = hexlify(encrypted_input) + hexlify(encrypted_key)
# 480 bytes(960 hex) + 128 bytes(256 hex) = 608 bytes(1216 hex)
```

### 2.2 RSA 公钥 ⚠️（GeeTest 换钥则全废）

`geetest_signer.py:22-24`，GeeTest 全局统一公钥（n 为 1024-bit，e=0x10001）。
**换钥自救**：`extract_rsa_key_from_browser()`（geetest_signer.py:27-60）——
先抓新的 `bcaptcha.js` 放到 `.cache/`，函数用正则 `,n=\[(214,144,233[^\]]+)\]`
提取字节数组，滑窗找 1024-bit 奇数 n。

### 2.3 e_obj 明文结构（w 的输入）⚠️ 最易随 GeeTest 构建漂移

`build_e_obj()`（geetest_signer.py:193-240），字段与顺序已与 2026-07-17
浏览器运行时实测对齐：

| 字段 | 值 | 说明/易变性 |
|---|---|---|
| `passtime` | 随机 1500–3500 ms | 模拟人工耗时 |
| `userresponse` | `[[round(x/300*10000), round(y/200*10000)] ×3]` | **坐标归一化到 0–10000**；底图尺寸 300×200 写死 ⚠️ |
| `device_id` | `""` | 空串 |
| `lot_number` | load 返回 | — |
| `pow_msg` / `pow_sign` | `f"{version}\|{bits}\|{hashfunc}\|{date}\|{captcha_id}\|{lot_number}\|\|{nonce}"` 的 md5 | `bits=0` 时一次命中；`bits>0` 才真算 PoW |
| `geetest` / `lang` | `"captcha"` / `"zh"` | 固定 |
| `ep` / `nqfq` / `EKAI` | `"123"` / `"622265669"` / `"y7R8"` | ⚠️ **gct4 构建常量**（`GCT4_STATIC`），当前构建 `gct4.614b49d4a6f9b9c251919ce8a63098bd`；换构建必须重提 |
| lot 动态键（如 `f275:{"4bf2":"e059"}`） | 从 lot_number 切片生成 | ⚠️ `LotParser`（geetest_signer.py:66-124），mapping 硬编码自 `__BOTION__.ctStore`，**botion 每次构建可能换 mapping** |
| `em` | `{"ph":0,"cp":0,"ek":"11","wd":1,"nt":0,"si":0,"sc":0}` | 固定 7 键（`EM_OBJ`） |

**已删除的旧字段**（平台演进痕迹，排查时不要加回来）：`biht`、`gee_guard`。

**诊断钩子**：`generate_w(..., diagnostics=dict)` 会回填 `e_obj_fields`（字段名列表）
和 `e_obj_bytes`（明文字节数），verify 失败时自动打印（http_login.py:449-458）——
真实浏览器 e_obj 约 476 bytes，**字段集变了 bytes 就变，这是构建漂移的直接观测点**。

## 3. 验证码答案来源（打码平台，无本地识别）

抽象接口 `CaptchaSolver`（`captcha_solver.py:143-171`），solver 链按序尝试、
全失败抛 `CaptchaSolveError`。**链顺序：geepass 优先，jfbym 备选**（http_login.py:81-87）。

| | geepass（首选） | jfbym（备选） |
|---|---|---|
| API | `POST https://api.geepass.cn/api/recognize/captcha` | `POST http://api.jfbym.com/api/YmServer/customApi` |
| type 码 ⚠️ | `30104` | `"31111"`（extra=`"je4_click"`） |
| 请求体 | `{token, type, image:<背景图b64>, ques:[3张字图b64]}` | `{token, type, image:<背景图b64>, image_label1/2/3:<字图b64>, extra}` |
| 成功响应 | `code==10000` → `data.data.targets` 边界框，取前 3 个算中心点 | `code==10000` → `data.data = "x1,y1\|x2,y2\|x3,y3"` |
| 繁忙重试 | `code==10009` → sleep 2s × 最多 3 次 | `code==10009` → sleep 3s × 最多 6 次 |
| token 来源 | `GEEPASS_TOKEN` 环境变量 | `JFBYM_TOKEN` 环境变量 |
| 速度/质量 | ~0.2s，无 again_tag | ~0.5s，常返 again_tag=2；坐标历史上有 ±20px 偏移 |

余额查询：jfbym `getUserInfoApi {token, type:"score"}` → `code==10001` → `data.score`。
其他非 10000 的 code = 余额不足 / token 错 / 题型不支持。

## 4. 站点 API 签名 X-API-XXX（与 w 完全独立，勿混淆）

| | GeeTest `w` | `X-API-XXX` |
|---|---|---|
| 用途 | botion verify（验证码答案包） | 乐鱼站点 API（/site/api、/game/api 请求签名） |
| 实现 | `geetest_signer.py` 纯 Python（RSA+AES） | `scripts/sign_wasm.cjs`（Node 跑官方 wasm）+ `api_sign.py` 封装 |
| 产物 | 1216 hex | 64 hex，每请求唯一 |

wasm 档案（`scripts/wasm_api_sign_bg.wasm`）：

- **来源** ⚠️：站点前端 `/_next/static/wasm/729ede6e9048bb61.wasm`
  （webpack 模块 87802 → 动态 chunk 2284，wasm-bindgen）。**站点前端重新部署
  可能换算法 → 需重新抓 wasm**；
- 初始化必须调 `wasm.exports[""]()`；imports 注入 `Date.now/Math.random/Math.floor`，
  所以每次签名不同属正常；
- 输入：`sign(path前缀, "prod")`，路径取前两段（`/site/api`、`/game/api`；
  `/component`→`/site/api`、`/page/fd`→`/fd/api`，api_sign.py:29-35）；
- Python `sign_path()` 校验输出必须 64 hex，依赖 `node` 在 PATH；
- Git Bash 需 `MSYS_NO_PATHCONV=1`；
- 兜底静态表：AES-128-CBC 解密 `uuidToBase64`，key=`ZFRYCMdFYGf0i5HgO0oWvFV0terUABU0`，
  IV=`CbE3P3t1lY34Ns8F`（token_manager.py:52-53, 693-701）；
- 服务端校验：kaptchcate 不校验；**/game/api 强制校验，假签名 → 6003**。

## 5. X-API-FINGER 指纹（仅 login 携带）

`hdata/auth/fingerprint.py`：fingerprintjs2 `x64hash128`（MurmurHash3 x64 128-bit，
**seed=31**）的纯 Python 复刻。

```
joined = f"24{1920},{1080}{timezoneOffset}0,false,false{user_ip}"
finger = x64hash128(joined, 31)   # 32 hex
```

- `user_ip` = validateGeeCheckV2 返回的出口 IP（服务端校验一致性）；
- 对拍真值：`x64hash128("241920,10804200,false,false219.76.134.210", 31)
  == "99c36b1529f2c9959a5d4aae2e19769f"`；
- ⚠️ **时区符号陷阱**（排查重点）：站点期望正数（实测串 `"420"`），`leyu_finger`
  默认 `timezone_offset=480`；但 `http_login._local_tz_offset()` 返回负数
  （UTC+8 机器 → -480）。符号不一致 → finger 与浏览器不符 → login 可能被风控拒。

## 6. 会话缓存与刷新

### `.cache/<账号>.json` 字段

```
game_token, game_player_id, game_backend, game_exp,
backend_domain_url_list, device_id,
domain, token, uuid, uuidToBase64, cookies, signatures,
source(cache/http_login/browser_login/...), updated_at, account
```

有效期唯一标准：`validate_game_token()` —— 游戏 JWT 剩余 >1h 才复用。

### `session.get_login()` 降级链（`session.py:491-651`）

1. 缓存 domain+token+有效 game_token → 直接返回；
2. 缓存有效但 game_token 过期 → `refresh_game_session()`（venue/launch 刷新）；
3. 有密码+打码 token → HTTP 打码登录全链路 → 再 venue/launch 补 game 字段；
4. 兜底 `browser_login`（可见浏览器人工/半自动登录）。

### 已知死路径/代码债（排查时别被误导）

1. `token_manager.py:493-596 _login_via_http` 是 http_login.py 的**旧拷贝**：
   无 kaptchcate、无公共头、无 X-API-FINGER，站点强制 wasm 签名后大概率已失效；
2. `token_manager.py:200-203` 硬编码的静态签名值（2026-07 前）仅作 setdefault，勿当真值；
3. `session.py:384` 读 `.session.json` 的 L1 路径**没有任何代码写这个文件**，实际不可达；
4. `captcha.py` docstring 里的 type 码（10114/30112）与实现（10118/31111）不一致，
   **以代码为准**；新链路只用它的 `CaptchaChallenge` 数据类，不调用 `captcha.solve()`。

## 7. 排查指南（平台更新后怎么办）

### 7.1 先看错误码/异常，定位环节

| 现象 | 环节 | 优先怀疑 |
|---|---|---|
| 入口站打不开/解析失败 | 步骤 0 | 域名轮换：`DomainCache().invalidate()` 重解析，或设 `HDATA_DOMAIN` |
| `TokenRefreshError stage=venue_launch*` | 步骤 8 | 域名失效 / X-API-XXX 签名（6003 = 签名错，重抓 wasm） |
| `stage=params_decrypt*` | 步骤 8 | params 解密规则变（key=ttl+"AES" 是否仍成立） |
| load 返回非 success | 步骤 2 | **captcha_id 换了** → 抓浏览器 load 请求取新值（3 处硬编码） |
| verify `result=fail` | 步骤 3 | w 解密正常只是坐标错 → 换打码平台/查余额；持续 fail 且换平台无效 → 归一化基准（底图尺寸）变了 |
| verify `unexpected_result`/`invalid_jsonp` | 步骤 2/4 | **w 本身坏了** → 走 7.2 标准动作 |
| `incomplete_seccode` | 步骤 4 | seccode 字段缺失，GeeTest 响应结构变了 |
| login 6003 | 步骤 6 | X-API-XXX wasm 更新 → 重抓 `/_next/static/wasm/*.wasm` |
| login 风控拒绝（非 6000 其他码） | 步骤 6 | X-API-FINGER：时区符号、user_ip 一致性 |
| `CaptchaSolveError code=10009` | 步骤 3 | 打码平台繁忙（自动重试中）；其他 code = 余额/token/题型 |
| WS 握手 HTTP 500 | 步骤 9 | 静态后缀版本 `v1.0.5` 变了 |

### 7.2 w 失效的标准排查动作（最重要）

前提认知：verify 返回 `fail` = w 能被正常解密（只是答案错）；返回其他异常 = w 结构坏了。

1. **抓真实 w**：跑 `scripts/capture_real_w.py`（Playwright + CDP 拦截真实浏览器
   的 botion/verify 请求，存 `data/captured_real_w_{i}.json`）；
2. **解密真实 w**：hook 浏览器 `Math.random` 拿 SDK 的 AES 随机密钥，解密出浏览器
   的真实 e_obj 明文（方法见 docs/captcha-flow.md §277-306）；
3. **diff 字段**：浏览器 e_obj vs `build_e_obj()` 输出——字段名、字段顺序、取值范围
   （真实 e_obj 约 476 bytes，bytes 数变 = 字段集变）；
4. **按 diff 修**：改 `GCT4_STATIC`（ep/nqfq/EKAI）、`LotParser.mapping`、`EM_OBJ`、
   字段顺序；
5. 若 `static.botion.com` 上 `gct4.<hash>.js` 文件名 hash 变了 = **第一件事重提构建常量**
   （从 `window.__BOTION__.ctStore` 提 LotParser mapping，从新 gct4.js 提 nqfq/EKAI）；
6. 若怀疑 RSA 换钥：抓新 `bcaptcha.js` 到 `.cache/`，跑 `extract_rsa_key_from_browser()`。

### 7.3 易变点速查表

| # | 易变点 | 当前值 | 位置 | 失效症状 / 自救 |
|---|---|---|---|---|

### 7.4 历史坑（都真实发生过，别再踩）

1. **w 长度变迁**：1568 hex → 1216 hex（勿按旧文档硬编码长度）；
2. **captcha_output 长度**：312 → 384 bytes（勿硬编码）；
3. **e_obj 字段漂移史**：2026-07-17 删 `biht`/`gee_guard`、增 `device_id/ep/nqfq/EKAI/em`，
   `userresponse` 改 0–10000 归一化；
4. **gct4 构建换代**：旧构建键是 `ZAhG:"MwHu"`，当前是 `nqfq/EKAI` 组合；
5. **X-API-XXX 静态表假设被推翻**：同端点同 body 多次签名全不同 → 实为 wasm 动态签名；
6. **curl_cffi 指纹被入口站 ban**（域名解析必须 urllib），但 botion/站点 API 反而都要
   `impersonate="chrome110"`；
7. **浏览器自动化坑**：Playwright `page.mouse.click()` 被极验 SDK 检测，必须 CDP
   `Input.dispatchMouseEvent`；headless 下验证弹窗可能不出现；
8. **打码精度敏感**：verify 对坐标极敏感，jfbym 历史上有 ±20px 偏移，持续 fail 先换 geepass 验证。
