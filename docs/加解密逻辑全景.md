# 加解密逻辑全景

> 面向接手者的完整加解密地图。读完本文不需要翻代码就能理解每一层
> 加解密的算法、密钥、流程；每层都标注代码位置（文件:行号），便于按图索骥。
> 协议细节（极验验证码、登录时序）另见《极验4验证码破解与排查.md》。
> 数据基于 2026-07-18 代码（commit d36f04b 之后）。

## 0. 总览：四层加解密 + 一套位图解码

| # | 层 | 通道/位置 | 算法 | 密钥 | 代码位置 |
|---|---|---|---|---|---|
| 1 | 登录 URL `params` | venue/launch 返回的游戏入口 URL | AES-128-ECB + PKCS7（无 gzip） | `ttl + "AES"`（动态，来自同一 URL） | `hdata/auth/params.py:28-78` |
| 2 | 站点 API 签名 `X-API-XXX` | `/site/api/*`、`/game/api/*` 请求头 | 官方 WASM 动态签名（黑盒，非加解密载荷） | 无密钥，wasm 内含时间+随机 | `scripts/sign_wasm.cjs`、`hdata/auth/api_sign.py` |
| 3 | WS 帧 | `wss://wsproxy.{backend}` 双向 | AES-128-CBC(gzip(JSON)) + PKCS7 | `ED7AA06BD8628B55`（key=iv） | `hdata/protocol/codec.py:36,49-63,99-111` |
| 4 | 网关 HTTP | `https://gateway.{backend}/game-http/*` | base64(AES-128-CBC(gzip(JSON))) + PKCS7 | `015CCB80A680E129`（key=iv） | `codec.py:70-93`、`hdata/client.py:1055-1101` |
| 5 | 路纸位图 | WS 推送的 `roadPaper.*` 字段 | base64 → MSB-first 位流切分（非加密） | 无 | `hdata/protocol/roadpaper.py` |

**三条最易踩的坑**（都真实踩过）：

1. WS 密钥是 **16 字节 ASCII 字符串**（`b"ED7AA06BD8628B55"`），不是 hex 解码的 8 字节；
2. WS 帧**没有任何二进制协议头**——整帧就是 AES 密文本身（旧文档 §3.2 的
   `[0x04][3B len][2B msg_id]` 假设已废弃，勿沿用）；
3. WS 与 gateway 算法同构但**密钥不同**，且 gateway 密文多一层 base64、
   请求多一套 HMAC-SHA1 签名头（`enc + "0" + timestamp`）。

## 1. 第一层：登录 URL `params` 解密（登录链路的起点）

venue/launch 接口返回 `data.url`，形如：

```
https://<游戏域>/?params=<base64>&ttl=<时间戳>&signature=...
```

解密（`hdata/auth/params.py:28-78` `decrypt_params`）：

- 算法：**AES-128-ECB + PKCS7**，key = `(ttl + "AES").encode()`
  （如 `"1782535308601AES"`，16 字节）；无 gzip；
- base64 容错：依次尝试 4 种候选（原样 / 空格转 `+` / URL 解码 / 两者组合，params.py:54）；
- 解出 JSON（即浏览器 `window.urlParams`）：

```json
{
  "playerId": 105452510,
  "token": "eyJhbG...（游戏 JWT，exp 约 24h）",
  "backendDomainUrl": "1j2cdazl.com:21027",
  "backendDomainUrlList": "1j2cdazl.com:21027,64vlwlq.com:18026,..."
}
```

- `token` = 游戏 JWT（WS 登录凭证）；`backendDomainUrl*` = 游戏后端域名（自带端口，动态轮换）。
- 有效期判断：`validate_game_token()`（params.py:112-131）——JWT `exp - now > 3600` 才算可用。

## 2. 第二层：站点 API 签名 `X-API-XXX`

> 详细排查指南见《极验4验证码破解与排查.md》§5，此处给全貌。

- 所有 `/site/api/*`、`/game/api/*` 请求必须带 `X-API-XXX: <64 hex>` 动态签名；
- 实现：Node 执行官方前端 wasm（`scripts/wasm_api_sign_bg.wasm`，来自站点
  `/_next/static/wasm/*.wasm`），封装在 `scripts/sign_wasm.cjs`；Python 侧
  `hdata/auth/api_sign.py:85-124` `common_headers()` 统一注入；
- 签名输入仅为**路径前两段**（如 `/site/api`、`/game/api`），与 body 无关；
  内含 `Date.now()` 与 `Math.random()`，**每次签名不同属正常**；
- 初始化必须先调 `wasm.exports[""]()`，否则 `unreachable`；
- 服务端校验强度：kaptchcate 不校验（假签名也 6022）；**`/game/api` 强制校验，
  假签名 → 6003 "非法请求"**；
- 兜底：wasm 失败时用 AES-128-CBC 解密 `localStorage.uuidToBase64` 静态签名表
  （key=`ZFRYCMdFYGf0i5HgO0oWvFV0terUABU0`，IV=`CbE3P3t1lY34Ns8F`，
  `hdata/auth/token_manager.py:52-53, 693-701`）；
- Git Bash 调用需 `MSYS_NO_PATHCONV=1`，否则 `/site/api` 被转成 Windows 路径。

公共请求头（`api_sign.py:85-124`）：`X-API-CLIENT: web`、`X-API-VERSION: 2.0.0`、
`X-API-SITE: 2001`、`X-API-UUID: <持久化大写 UUID4>`、`X-API-TOKEN: <96 hex>`、
`X-API-XXX: <64 hex 动态>`、`X-API-FINGER: <32 hex，仅 login 携带>`。

## 3. 第三层：WS 帧协议（游戏数据主通道）

### 3.1 帧 = 纯密文，无二进制头

发送/接收的 WS binary frame 载荷**整帧就是 AES 密文**，没有魔数/版本/长度/标志位。
WS 文本帧（`isinstance(raw, str)`）直接丢弃（`hdata/client.py:551-555`）。

### 3.2 算法（`hdata/protocol/codec.py`）

```python
AES_KEY = b"ED7AA06BD8628B55"   # codec.py:36，游戏前端 DataHandle._defaultKey 硬编码
```

- **AES-128-CBC**，**IV = KEY**（`modes.CBC(AES_KEY)`，codec.py:53,60）；
- **PKCS7** 填充（16 字节块，codec.py:51-52, 63）；
- **gzip + JSON 紧凑序列化**：发送 `gzip(JSON) → AES`；接收 `AES 解密 → gunzip → JSON`：

```python
def encode_frame(msg: dict) -> bytes:            # codec.py:99-102
    plaintext = gzip.compress(json.dumps(msg, separators=(",", ":")).encode())
    return aes_encrypt(plaintext)

def decode_frame(raw: bytes) -> dict | None:     # codec.py:105-111
    try:
        data = aes_decrypt(raw)
        return json.loads(gzip.decompress(data).decode("utf-8"))
    except Exception:
        return None                               # 任何一步失败返回 None
```

收发双向同一把 key/iv/算法。

### 3.3 消息结构与签名（`codec.py:117-144` `build_message`）

外层帧字段：

| 字段 | 说明 |
|---|---|
| `jsonData` | `JSON.stringify({"id": protocolId, "param": JSON.stringify(data)})` —— **param 是二次字符串化的 JSON** |
| `nonce` | `random.randint(0, 2**31)` |
| `sign` | `Base64(HmacSHA1(jsonData + nonce + timestamp, key=AES_KEY))`，拼接顺序固定：jsonData → nonce → timestamp，无分隔符 |
| `timestamp` | 毫秒时间戳 |
| `protocolId` / `gameTypeId` / `playerId` / `tableId` / `serviceTypeId` | 见常量表 |

解包（`codec.py:171-189` `extract_param`）：外层 → `jsonData`（str 则再 `json.loads`）
→ 取 `param`（同理）→ 返回 `{"id","param","status","msg","data"}`。
消费侧统一写法 `payload = info.get("param") or info.get("data")`。

常量（codec.py:38-43 + client.py:60-69）：

```python
FS_LOGIN = 10000        # 登录请求/响应
FS_LOGIN_FAIL = 10026   # 会话级踢出（token 失效/被顶替）
OT_HALL = 7             # serviceTypeId 大厅
OT_GAME = 3             # serviceTypeId 游戏
DEVICE_TYPE_PC = 15     # PC 网页端
# client.py 内部：
_QS_TABLE_LIST_ALL = 10089   # 拉大厅桌台（data={"labelTypeId":1}）
_QS_NEW_INTER_GAME = 401     # 普通桌进桌
_QS_INTER_GAME = 101         # VIP/竞价等进桌（gameTypeId ∈ {2003,2004,2014,2020}）
_QS_OUT_GAME = 102           # 离桌（请求）/ 离桌推送（响应）
_QS_NOTICE = 123             # 系统通知推送（预警）
_HT_SEAT = 1                 # identity：座位
_PT_BASE = 2                 # joinTableMode：基础
```

### 3.4 WS 连接与登录握手

连接 URL（`hdata/auth/session.py:433-481` `build_ws_config`）：

```
wss://wsproxy.{backendDomainUrl}/?playerId={pid}&jwtToken={游戏JWT}&deviceId={deviceId}
     &platformId=1&applicationId=5&version=v1.0.5
```

- `backendDomainUrl` 自带端口，host 前加 `wsproxy.` 子域；
- **静态后缀缺失 → 握手直接 HTTP 500**（session.py:47-51）；
- `deviceId` 格式：`{13位毫秒}{6位随机}-{8位随机}`（session.py:54-67，复刻浏览器生成逻辑）；
- **每次新建连接前先刷新 game_token**：服务端按 JWT jti 做单连接会话管理，
  旧 token 即刻作废（client.py:510-516 `on_before_connect` 钩子）。

登录（codec.py:147-168 `build_login_msg` → client.py:530-546 `_login`）：

- 发送 `protocolId=10000`，data 含 `jwtToken/deviceType=15/deviceId/timeZoneArea/
  offsetMinutes/protocolCodecConfig/version="1.1.1"`；
- 成功 = 收到 `protocolId=10000` 且 `extract_param()["status"] == 1`；
- 12 秒超时；期间收到 10026 = token 已被顶替/失效。

## 4. 第四层：网关 HTTP（game-http/*，玩家设置读写）

与 WS 帧同构，唯二区别：**密钥不同、密文再套 base64**。

```python
GATEWAY_KEY = b"015CCB80A680E129"      # release 环境（codec.py:70）
GATEWAY_KEY_DEV = b"AA4194657AD89A56"  # dev/training 环境
```

- `gateway_encrypt`（codec.py:74-83）：`dict → gzip(JSON 紧凑) → PKCS7 → AES-CBC → base64`；
- `gateway_decrypt`（codec.py:86-93）：`base64 → AES-CBC 解密 → 去填充 → gunzip → dict`，
  base64 自动补 `=` 容错。

请求构造（`hdata/client.py:1055-1101` `_gateway_request`）：

- 关键头：`deviceType: "15"`、`model: <Chrome UA>`、`deviceId`、`X-Request-Token: <game_token>`、
  `keyid: "probinpjms7rfm26"`（release，大厅 bundle 硬编码；⚠️ 它不是载荷密钥）；
- POST 签名链（GET 无签名）：
  1. `enc = gateway_encrypt(payload)`（同时就是请求 body）；
  2. `sign = Base64(HmacSHA1(enc + "0" + str(timestamp), GATEWAY_KEY))`
     —— 拼接：密文 → 字面量 `"0"`（nonce=0）→ 毫秒时间戳；
  3. `meta = {"encrypted":true,"gzipped":true,"platform":"h5","version":"1.2.2",
     "application":"game_http","timestamp":ts,"nonce":0,"sign":sign,"keyid":"probinpjms7rfm26"}`；
  4. `headers["token"] = gateway_encrypt(meta)` —— meta 本身也用同算法加密；
- 响应：先尝试 `json.loads`（明文），失败再走 `gateway_decrypt`（密文），两种兼容；
- HTTP 客户端：`curl_cffi`，`impersonate="chrome110"`；
- 已用端点：`GET/POST https://gateway.{game_backend}/game-http/player/getPlayerSetting`
  与 `updatePlayerSetting`（设置读写，含路纸筛选 `settingObject="23"`）。

## 5. 路纸位图解码（非加密，是自定义位打包）

`hdata/protocol/roadpaper.py`，与游戏前端 `_i` 类一致。

### 5.1 位读取器（roadpaper.py:37-58）

`base64 → 字节流 → 每字节 8 位二进制串拼接 → MSB-first（高位在前）顺序消费`；
越界读补 0 容错。

### 5.2 数据布局

```
头部:  version = read(8) + 1                # 注意要 +1
       总格数  = read(8) * read(8)           # 两个 8 位相乘
每格:  flag = read(1)
       大路族(bigRoad):    flag=1 → result=read(2), tieNumber=read(4)   # 共 7 bit
       珠盘族(beatPlate):  flag=1 → result=read(2), pair=read(2)        # 共 5 bit
       flag=0 → 空位（None，仍占一格）
分列:  每 6 格一列（COLUMN_SIZE=6，roadpaper.py:34）
```

结果枚举（roadpaper.py:22-32）：`0=P 闲 / 1=B 庄 / 2=T 和 / 3=B6 幸运6庄`。

### 5.3 路纸键与解码器映射（roadpaper.py:132-152）

```python
ROAD_DECODERS = {
    "bigRoad": decode_big_road, "notOutBigRoad": decode_big_road,
    "winLoseRoad": decode_big_road,
    "beatPlateRoad": decode_bead_plate, "beatPlateRoad2": decode_bead_plate,
}
```

- 只有两族格式；`bigEyeBoy`/`smallRoad`/`cockroachPig` 等其余键**未实现解码**，
  `decode_road_paper` 遇未知键直接跳过；
- 常用入口：`decode_bead_plate(b64)["flat"]` → 按顺序的非空 B/P/T/B6 序列
  （`MultiTableSession.road_flat()` 即此）；
- 连胜口径：`client.py` `road_streak()` —— T 归前一非和胜方、不断连胜不占局数，B6→B。

### 5.4 RoundTracker（`hdata/protocol/round_tracker.py`）

路纸/结果的增量状态层：`feed_road_paper` 用珠盘 flat 与历史对齐、只追加新局（幂等），
`feed` 在 `streak_count >= 5` 时产出长龙信号。

## 6. 协议号清单（现行代码权威口径）

| protocolId | 方向 | 含义 | 关键字段/说明 |
|---:|---|---|---|
| 10000 | 请求/响应 | WS 登录 | 响应 `status=1` 成功 |
| 10026 | 推送 | 会话级踢出 | token 失效/被顶替（jti 单连接），须重新 login |
| 10089 | 请求 | 拉大厅桌台 | `data={"labelTypeId":1}`，serviceTypeId=7 |
| 10052 | 推送 | 大厅快照 | `gameTableMap{tableId:{gameTypeId,gameStatus,bootNo,roadPaper,tableOnline,...}}`，分批增量 |
| 401 | 请求/响应 | 普通桌进桌 | 响应含 `gameTableInfo` 全量快照 |
| 101 | 请求/响应 | VIP/竞价等进桌 | gameTypeId ∈ {2003,2004,2014,2020} |
| 102 | 请求 | 主动离桌 | 空 body |
| 102 | 推送 | 离桌通知 | `leaveTableType=1` 主动确认(noticeId=21001)；`=2` 未下注被踢(noticeId=21003) |
| 123 | 推送 | 系统通知 | 连续 3 局未下注预警(noticeId=21002)，**只预警不踢人** |
| 104 | 推送 | 局状态 | `roundNo/countdown/bootIndex` → 事件 type="round" |
| 106/107 | 推送 | 发牌/牌局事件 | → type="card" |
| 110 | 推送 | 桌台动态 | roundId/在线/投注/奖池 → type="bet" |
| 116 | 推送 | 路纸 | 含 `roadPaper` 各键 base64 位图 → type="road" |
| 160/161 | 推送 | 路纸更新 | → type="road" |
| 171 | 推送 | 桌台状态 | → type="status" |

踢出机制（2026-07-18 四轮在线实验结论，commit d36f04b）：

- 连续未下注满 **3 局** → 123 预警（不踢人）；满 **5 局** → 102 推送（`leaveTableType=2`）真踢，
  **WS 连接不断**，只收不到该桌数据；
- **被踢前重发 401 不能防踢**（实测不重置计数）；唯一策略是被踢后立即重进（7 秒内恢复）；
- `kick_policy="stay"`（默认）自动重进并产出 `type="kick"` 事件；`"follow_system"` 停止监控该桌；
- 进桌时进行中的那一局也计入未下注局数（自该局结算起计）。
