# 代理接入

> 目的：突破单出口 IP 的 WS 连接速率/并发限制，按 IP 线性放大
> 监控容量；同时分散单 IP 被封禁的风险。
>
> 状态：P0~P3 已完成并通过全链路活测（2026-07-20，见 §6）；
> P4 运维化（运行中掉线自动换绑）待实施。

## 1. 容量模型

实测工作点（详见《平台接入机制》4.4 节）：

- 单出口 IP：3s 间隔建连可稳定维持 **~11 条 WS 并发**
  （1 大厅 + 10 监控）；单连接可监控 11+ 张桌
- 即 **1 个 IP ≈ 110+ 桌监控容量**；N 个代理 IP 线性扩容
- 全平台约 200 桌：2~3 个 IP 即可全覆盖，更多 IP 用于冗余

注意：并发硬上限未知（11 条是账号数天花板而非平台实锤上限，
见 §6 待办），规划时按 11 条/IP 的安全工作点计算。

## 2. P0 探针结论（2026-07-19，`scripts/probe_proxy.py`）

| 验证项 | 结论 |
|---|---|
| 代理可用性 | 无忧 HTTP 代理（账密认证）出口正常，延迟 0.2~0.3s |
| websockets 兼容 | 16.0 `proxy=` 参数对 HTTP CONNECT 账密代理兼容良好 |
| **token 绑 IP** | **实锤绑定**：对照实验同一新鲜 token，直连 WS 成功、代理 WS 立刻 10026 |

绑 IP 对照实验方法：强制全量重登取新鲜 token → A 组直连 WS
（证明 token 有效）→ B 组同一 token 走代理 WS（10026 被拒）。
排除"缓存 token 过期"干扰（这是此前误判的常见原因）。

## 3. 核心设计原则：账号 ↔ 代理粘性绑定（硬性要求）

由于 token 绑定登录 IP：

1. 每个账号的**登录、token 刷新、WS 连接全程走同一代理出口**
2. 多个账号可共享同一代理 IP（连接额度按 IP 计），但同一账号
   不可跨代理漂移
3. 打码平台 API（geepass/jfbym）**不走代理**——第三方服务，
   与平台 IP 无关
4. 域名解析（种子站→活域名）集中做一次，各代理共用结果
5. **提供了代理就只用代理**（2026-07-20 用户决策）：配置了代理
   文件时本机直连**不**作为出口混入，避免误用直连顶号；
   仅在完全未配置代理时全体走本机直连。代理全部失效时
   程序报错退出，不静默回落直连
6. **只有 game_token 绑 IP，站点会话（cookies）不绑**
   （2026-07-20 实测，见《平台边界试探》4.4）：缓存会话命中时
   经新代理出口刷新一次 game_token 即完成换绑，**无需**删缓存
   强制打码重登

## 4. 出口分配策略（P2 已实现，`hdata/proxy.py`）

把"出口"抽象为资源池，**每出口一份连接预算**（默认 10 条，
实测安全工作点，硬上限压测前不建议调大）：

1. 发现层账号先分，监控账号**均衡轮询**到负载最小的存活出口
2. 账号↔出口**粘性**：分配一次后进程内不变；代理过期换文件后
   重启自然重新分配
3. 总容量不足（账号数 > 出口数 × 预算）：超出账号**告警弃用**，
   日志点名；发现层账号无出口则直接退出
4. 代理中途失效：`mark_dead` 摘除 → 名下账号解绑重分 →
   **经新出口重新登录**拿新 token（token 绑 IP 决定的必然代价），
   继续监控；出口跳变记入《平台边界试探》观察风控
5. 代理多、账号少：自然稀疏分配（一账号一出口优先），
   多余出口作故障切换备用池

## 5. 代理选型

- **协议**：HTTP（CONNECT 隧道）即可——全部流量本身 TLS 端到端
  加密，SOCKS5 无实际收益且 WS 层兼容性需额外验证
- **类型**：静态住宅/ISP 代理（粘性会话，单 IP 固定数小时~数天）；
  避免分钟级轮换代理（WS 长连必断）与免费代理
- **带宽**：4 Mbps 足够（单 IP 满载 ~0.3 Mbps，10 倍余量）；
  真正要核的是**流量额度**——单 IP 满载 24h ≈ 3~4 GB/天 ≈ 100 GB/月
- 凭证存放：`data/proxies.json`（已被 gitignore，不进版本库）
