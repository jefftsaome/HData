# 游戏平台接入机制

> 本文档梳理 HData 接入游戏平台的完整机制：域名解析 → 登录 → 会话 → WS 数据通道 → 桌台数据。
> HData 按**平台无关**设计：`hdata/client.py` 的 `GameClient` 是对外唯一门面，
> leyu（乐鱼）是当前已接入的子平台；接入新平台时按本文档的机制复刻适配层即可，
> 对外接口保持不变。

## 0. 总体架构

```
调用方
  │  GameClient(entry_url).login() / get_tables() / enter_table()
  ▼
┌─────────────────────────── hdata ───────────────────────────┐
│  client.py        对外门面（平台无关，稳定签名）              │
│  auth/            平台适配层：域名解析、登录、验证码、会话      │
│  protocol/        协议层：帧编解码、加解密、路纸解码            │
│  capture/         通道层：WS 连接、CDP 桥接（调试用）          │
│  sources/         htools DataSource 接口实现                  │
└──────────────────────────────────────────────────────────────┘
  │
  ▼
平台基础设施：种子站 → 动态域名 → API → WS 代理 → 游戏后端
```

平台共性（所有子平台都适用的规律）：

1. **域名动态轮换** — 真实域名/端口小时级失效，但种子站固定
2. **验证码登录** — 极验点选验证码，需打码平台
3. **双 token 体系** — 站点 session token + 游戏 JWT，后者按 jti 单连接
4. **加密 WS 帧** — AES 加密 + gzip + JSON 的自定义帧协议
5. **两级数据** — 大厅快照（桌台列表）与桌内快照/事件流
6. **踢出机制** — 桌台级（不投注被踢，可重进）与会话级（token 失效，需重登）

## 1. 域名解析机制

**问题**：平台真实域名（如 `www.xxxx.vip:9003`）与后端地址动态轮换，小时/天级失效。

**解法**（`hdata/auth/domain.py`）：

1. 调用方只提供**种子站**（如 `https://leyu.com` / `https://leyu.me`），种子站长期有效
2. 抓种子站的 `/code.js`，其中的 `lypcurls='...'` 字段是 PC 端当前真实域名列表
3. 对候选域名**探活**，取第一个可用的
4. 结果缓存 TTL 30 分钟，失效自动重新解析

调用方永远不缓存、不硬编码任何真实域名。

## 2. 登录机制

三级降级（`hdata/auth/session.py` 的 `get_login`，内部自动选择）：

| 优先级 | 方式 | 成本 | 说明 |
|:------|:-----|:-----|:-----|
| 1 | 缓存会话 + 刷新 | 0 | `.cache/{账号}.json` 有效时直接用；game_token 过期则 HTTP 刷新 |
| 2 | 纯 HTTP 打码登录 | 打码费 | `http_login.py`，无浏览器，需提供打码平台 token |
| 3 | 浏览器辅助登录 | 人工 | Playwright 兜底，人工点验证码 |

### 2.1 纯 HTTP 登录流程（`hdata/auth/http_login.py`）

```
获取验证码背景/问题图 → 打码平台识别点选坐标 → 生成 w 参数（RSA+AES加密）
→ verify 校验 → 从 seccode 提取 captcha_output/gen_time/pass_token
→ validateGeeCheckV2 → login 接口 → 站点 token
```

- 打码平台双通道：geepass（优先）/ jfbym（备选），各自独立 token
- 密码 MD5 后传输；`w` 参数生成见 `geetest_signer.py`（RSA 公钥从 `.cache/bcaptcha.js` 静态数组提取）

### 2.2 登录产物

站点 token、用户 UUID、当前域名、cookies、API 签名表 —— 全部写入会话缓存。

## 3. 会话与 game_token 机制

进入游戏数据通道需要第二张凭证：**游戏 JWT**（`game_token`）。

- 用站点 session 调 `refresh_game_session` 换取，含 player_id、backend、过期时间
- **jti 单连接语义（关键）**：服务端按 JWT 的 jti 做会话管理，
  **同一账号同时只允许一条游戏连接**；新登录/刷新后旧 token 即刻作废
- 因此：**每次新建 WS 连接前必须先刷新 token**，不能信任缓存里的旧 token
  （`GameClient` 内部已内置此行为）
- token 非一次性：有效期内可复用，但必须是服务端当前认可的那张

## 4. WS 数据通道机制

### 4.1 连接

```
wss://wsproxy.{backend:port}/?playerId=..&jwtToken=..&deviceId=..
    &platformId=1&applicationId=5&version=v1.0.5
```

query 参数缺一不可（缺后缀握手返回 500）。backend 来自 game_token 刷新响应。

### 4.2 帧协议（`hdata/protocol/codec.py`）

```
整帧 = AES-128-CBC( gzip( JSON ) )，key = iv（平台常量）
```

- 登录帧：protocolId=10000（Fs.Login），serviceTypeId=7（大厅）
- 游戏消息：serviceTypeId=3
- 编解码对外不可见，`GameClient` 内部处理

### 4.3 心跳与重连

协议心跳帧保活；连接断开按域名列表轮询重连。

## 5. 桌台数据机制（两级）

### 5.1 大厅快照（`get_tables`）

发 10089 请求 → 收 10052 增量推送 `gameTableMap`，每张桌含：

| 字段 | 说明 |
|:-----|:-----|
| gameTypeId | 游戏类型（2001=经典百家乐 等） |
| gameStatus | 状态：2 下注中 / 3 发牌中 / 4 开牌结算 |
| bootNo | 靴号 |
| roadPaper | 路纸（base64 位串，珠盘解出 B/P/T/B6 序列） |
| tableOnline | 在线人数 |

⚠️ 大厅快照**没有 tableName**，桌名要进桌后才有。

### 5.2 桌内数据（`enter_table`）

- 进桌协议号：普通桌 401（NEW_INTER_GAME）；VIP/竞价等类型（2003/2004/2014/2020）用 101
- 401 响应的 `gameTableInfo` 是全量快照：tableName、dealerName、roundNo、cardResult、限额等 60+ 字段
- 之后持续推送事件流：104 局状态 / 106·107 发牌 / 110 桌台动态 / 116·160·161 路纸

### 5.3 路纸解码（`hdata/protocol/roadpaper.py`）

base64 → MSB 位串 → 珠盘（beatPlateRoad）出 `B`庄/`P`闲/`T`和/`B6`幸运6庄 序列。

## 6. 踢出机制（实测）

连续未下注的完整生命周期（2026-07-18 四轮在线实验确认）：

| protocolId | 级别 | 触发 | 负载关键字段 | 处理 |
|:-----------|:-----|:-----|:-------------|:-----|
| 123 | 通知 | 连续 **3** 局未下注 | `{"message": "您已连续3局未下注，后2局未下注将退出游戏", "noticeId": 21002}` | 仅预警，不踢人；作为 `notice` 事件透出 |
| 102 | 桌台级 | 连续 **5** 局未下注 | `{"tableId": …, "leaveTableType": 2, "msg": "您长时间未下注，已退出游戏", "noticeId": 21003}` | 真踢出，**WS 连接不断**；自动重进该桌即可恢复（`TableSession`/`MultiTableSession` 已内置） |
| 102 | 桌台级 | 自己主动发 102 离桌 | `leaveTableType: 1, noticeId: 21001` | 离桌确认推送，内部自动忽略 |
| 10026 | 会话级 | token 失效/被顶替 | — | 重新 `login()` 刷新凭证 |

关键结论：

- **无法防踢**：被踢前重发进桌指令（401）**不能**重置未下注计数（实测重发后仍按原节奏被踢），
  且入座中重复 401 可能引起连接异常。唯一可靠策略是被踢（102 推送）后立即重进。
- 进桌时进行中的那一局也算入未下注局数（从该局结算起计）。
- 被踢后重进即时生效，后续局事件恢复正常推送。

## 7. 对外接口速览

```python
from hdata.client import GameClient

client = GameClient(entry_url="<平台种子站>",
                    geepass_token="...", jfbym_token="...")
session = await client.login(account, password)   # → 会话凭证
tables  = await client.get_tables()               # → 桌台列表
async with await client.enter_table(table_id) as ts:  # → 进桌
    ts.snapshot                                   #   桌内快照
    async for ev in ts.events(): ...              #   事件流
```

## 8. 接入新子平台清单

接入一个新平台时，按序确认以下六点（对应本文档 §1–§6）：

1. ☐ 种子站 URL；`/code.js`（或等价物）中真实域名字段名
2. ☐ 登录接口、验证码类型与 `w` 参数加密方式
3. ☐ 站点 token → 游戏 token 的换取接口；是否 jti 单连接
4. ☐ WS 代理 URL 模板、帧加密算法与密钥
5. ☐ 大厅/桌内两级数据的协议号与字段结构
6. ☐ 踢出规则与心跳间隔

确认后实现 `auth/` 适配层 + `protocol/` 编解码即可，`GameClient` 对外签名不变。
