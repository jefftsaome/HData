# 招聘技术指南

## 沟通顺序（重要：不要跳步）

```
Step 1: 聊技术方向 → 看对方有没有底线，能不能系统思考
Step 2: 沟通技术底线 → 确认边界一致
Step 3: 聊具体行业 → 看对方是否接受
Step 4: 深入技术细节 → 验证实战能力
```

---

## Step 1：技术方向概述（可以公开）

> 我们在做一个实时数据平台，整体架构是：
>
> **数据采集层** — 多源接入（WebSocket 实时流 + DOM 轮询 + HTTP API），核心瓶颈是目标网站的自动化登录，涉及浏览器指纹绕过、验证码识别、API 签名逆向。
>
> **数据处理层** — 协议解码、数据清洗、事件标准化，把多源异构数据转成统一的 MarketTick 格式。
>
> **策略引擎层** — 实时策略评估，插件式接入用户策略，聚合并行结果。
>
> **数据存储层** — 时序数据 + 路纸历史，加密存储。
>
> **面向未来的设计** — 整个系统按 monorepo 组织，共享接口库 + 独立实现模块，通过 entry_points 插件发现，新增数据源或策略只需要实现接口、注册即可。
>
> 当前 Python 3.13 + asyncio，正在从单体架构向插件化框架迁移。
>
> 你之前在数据平台或者分布式系统方面有什么经验？

### 这段在聊什么

不是让对方做题，是看对方能否理解系统全貌——不只是"写个爬虫"，而是**数据从哪来、怎么变、往哪去、怎么保证不掉数据**。

如果对方只问"爬哪个网站"而不问架构——这个人视野可能偏窄。
如果对方问"数据一致性和延迟怎么保证"——这个人有系统思维。

---

## Step 2：沟通技术底线（必须，在暴露行业之前）

> 我们的数据源是境外网站，技术上涉及一些灰色地带——绕过验证码、浏览器指纹对抗、协议逆向。这些都在数据采集的只读层面，不涉及任何交易操作。
>
> 我想先确认一件事：你对这种技术方向有没有底线？有没有哪些事你觉得绝对不能做？

### 怎么判断

| 对方反应 | 判断 |
|---------|------|
| "我什么都做，没底线" | ❌ 不能要 — 不是能干什么，是没有判断力 |
| "博彩/灰产类的不碰" | ⚠️ 有底线 — 追问"如果只做只读数据采集呢？" |
| "只读采集没问题，只要不碰资金/不攻击服务器" | ✅ 边界清晰 |
| 反问"具体什么网站？什么数据？" | — 不要直接回答，说"稍后会说明，先聊你的底线" |

---

## Step 3：聊具体行业（对方通过 Step 2 之后）

> 我们的目标网站是一个境外博彩平台。我们做的是实时盘口数据采集和策略分析，只读数据，不涉及下注、不篡改数据、不对服务器做任何攻击。
>
> 团队很小，每个人对结果负责。你需要独立分析问题、设计方案、推进实施。
>
> 你对这个方向有没有顾虑？如果没问题我们继续聊技术细节。

---

## Step 4：深入技术细节

### 可以给候选人看的技术挑战

```
数据采集层当前最大瓶颈：目标网站的自动化登录。

反爬链路：
  HTTP 444（非浏览器拒绝连接）
  → 浏览器指纹检测（WebGL/GPU/PluginArray/WebDriver）
  → 文字点选验证码（GeeTest v4类型，坐标点击）
  → API 请求签名（AES-CBC加密的路径→签名映射表，新域名全为空）

当前方案：
  L0-L3 四层降级的 token 获取：
    L0: 缓存命中 → 0s 返回
    L1: session 有效 → API 刷新
    L2: 浏览器 session 有效 → 自动跳转截获
    L3: 完整登录 → stealth browser + 底层协议 + 打码平台

  为什么用底层协议而不是 Playwright click：
    上层自动化工具的鼠标事件被验证码 SDK 检测为自动化
    底层协议直达浏览器输入层，事件属性与真实操作无异

已知脆弱点（共 9 项，分 🔴🟡🟢 三级）
  需要候选人能自己分析出至少 5 项

面向未来：
  - 多源数据接入的插件化架构
  - 数据协议解码 + 标准化 pipeline
  - 策略引擎的实时评估框架
```

### 面试题三道

**问题 1：逆向深度**

> 我们的点击方案用了底层浏览器协议直接发送鼠标事件，而不是上层自动化工具的 click。你觉得为什么前者能绕过检测？如果 SDK 升级，可能从哪几个方向检测？

考察点：浏览器事件模型（event.isTrusted、事件链路、timestamp 精度）、JS 层 vs 协议层的检测差异、能否提出具体检测向量。

**问题 2：系统思维**

> 这是当前的架构和已知脆弱点。你觉得还有遗漏吗？如果让你来负责，加固优先级怎么排？

考察点：能否跳出单个模块看到系统全貌；优先级判断是否务实；是否有"先确保可用再逐步加固"的工程思维。

**问题 3：实战能力**

> 给你一个新场景：同一个验证码类型，换了一个网站。一周内跑通。你会从哪几个方向入手？每一步可能遇到什么坑？打码平台挂了怎么办？

考察点：方法论是否清晰（观察→拆解→最小复现→规模化）；对第三方依赖是否有备选；时间评估是否务实。

---

## 评估标准

| 维度 | 及格 | 优秀 |
|------|------|------|
| 逆向深度 | 能说出 bot 检测的基本原理 | 能提出具体检测向量和绕过思路 |
| 系统思维 | 能看到显式风险 | 能发现遗漏风险 + 排能落地的优先级 |
| 实战能力 | 有方法论但细节模糊 | 步骤清晰 + 备选方案 + 时间评估 |
| 架构视野 | 只看当前模块 | 能提数据一致性、监控、扩展性 |
| 沟通 | 回答问题 | 能提好问题 + 主动沟通边界 |

## 加分信号

- 主动问"数据对不上的时候怎么处理的"
- 主动问"如果 stealth browser 跑路了有没有替代方案"
- 说"这个我做过类似的，坑在 XXX"而不是"这个很简单"
- 对行业有顾虑但能理性讨论而不是一刀切拒绝

## 红线信号

- 聊到具体技术时追问"什么网站"超过三次（底线沟通前不应追问）
- "我什么都做过"但说不清一个具体问题的解决方案
- 对技术底线问题表示"无所谓"/"我不管这些"
- 主动提议"可以顺便搞点别的"（DDoS、刷量等）
